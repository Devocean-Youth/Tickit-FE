## 📌 브랜치 전략

### 브랜치 이름 규칙
`[type]/[issue-number]-[short-description]`

| 타입 | 설명 |
|------|------|
| `feat/` | 새로운 기능 추가 |
| `fix/` | 버그 수정 |
| `refactor/` | 리팩토링 |
| `docs/` | 문서 관련 |
| `chore/` | 설정, 패키지 등 |

**예시**:
- `feat/23-login-feature`
- `fix/45-token-bug`

### 브랜치 생성 예시
```bash
git checkout -b feat/23-login-feature
```

## 🗂️ 이슈 작성 규칙

### 이슈 제목 예시
- `Feat: 로그인 기능 구현`
- `Fix: 로그인 상태가 저장되지 않는 문제 수정`

### 이슈 본문 템플릿
```md
## ✨ 설명
- 어떤 기능 또는 문제인지 간단히 설명

## 📌 작업 항목
- [ ] 작업 1
- [ ] 작업 2

## 🔗 관련 링크
- 디자인 링크, API 문서 등
```

---

## 💬 커밋 메시지 컨벤션
```
[type]: 작업 요약
```
- 예: `feat: 로그인 API 연동`

| 타입 | 설명 |
|------|------|
| feat | 기능 추가 |
| fix | 버그 수정 |
| refactor | 리팩토링 |
| docs | 문서 작성 |
| chore | 기타 작업 |

---

## 🙋‍♀️ 협업 규칙 요약
- 이슈 먼저 생성 후 브랜치 생성
- 하나의 브랜치 = 하나의 작업 단위
- PR 생성 시 이슈 번호 명시 (ex: `close #23`)
- 머지 전 리뷰어 1명 이상 승인 필요 (옵션)
